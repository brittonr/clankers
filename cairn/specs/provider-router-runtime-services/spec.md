# Provider Router Runtime Services Specification

## Purpose

Defines the `provider-router-runtime-services` capability.

## Requirements

### Requirement: Runtime provider model contract is neutral [r[provider-router-runtime-services.model-contract]]

The runtime provider/router service MUST expose neutral model request, stream, response, usage, and failure DTOs suitable for engine-host execution without requiring provider-native `CompletionRequest`, router RPC frames, OAuth store types, or provider body JSON at the SDK boundary.

#### Scenario: Neutral request covers engine needs [r[provider-router-runtime-services.model-contract.neutral-request]]
- GIVEN the engine-host path needs model execution
- WHEN it constructs a runtime provider request
- THEN the request MUST contain neutral model label, session id, system prompt, messages/content, tools, thinking/cache options, and safe metadata
- AND it MUST NOT expose provider-native request/body types, daemon frames, TUI state, OAuth store internals, or router daemon handles

#### Scenario: Streamed outcome is first-class [r[provider-router-runtime-services.model-contract.streamed-outcome]]
- GIVEN a provider streams assistant text, thinking, tool-use JSON, usage, or errors
- WHEN the runtime provider service returns output
- THEN it MUST provide semantic stream events or a completed neutral response that can feed engine-host accumulators
- AND receipt-only byte counts MUST NOT be the only model execution result

### Requirement: Provider auth services are host-owned [r[provider-router-runtime-services.auth-contract]]

Auth lookup, credential-pool selection, token refresh persistence, pending login verifier storage, and account scoping MUST be supplied by explicit host services and MUST fail closed when unavailable in embedded mode.

#### Scenario: Auth operations are explicit [r[provider-router-runtime-services.auth-contract.host-owned]]
- GIVEN model execution needs credentials, refresh, login state, or account selection
- WHEN the runtime asks for that behavior
- THEN it MUST call a host-owned service with safe request and receipt DTOs
- AND it MUST NOT read or mutate desktop auth files, pending login verifier files, router cache DBs, or environment credentials unless a desktop adapter was explicitly injected

### Requirement: Desktop provider adapter delegates policy [r[provider-router-runtime-services.desktop-adapter]]

Desktop adapters MAY bridge runtime provider services to current `clankers-provider` and `clanker-router`, but they MUST delegate provider-native request shaping and routing/fallback/cooldown policy to the existing owners.

#### Scenario: Desktop adapter does not duplicate provider/router policy [r[provider-router-runtime-services.desktop-adapter.delegates-policy]]
- GIVEN the desktop adapter handles a neutral provider request
- WHEN it invokes Clankers provider/router code
- THEN provider-native body construction MUST occur in the provider/router owner module
- AND routing, fallback, cooldown, refresh retry, and known-provider fail-closed behavior MUST remain owned by router/provider policy modules

### Requirement: Provider service verification uses literal fixtures [r[provider-router-runtime-services.verification]]

Verification MUST pin request/response/auth contracts with deterministic fixtures and parity tests.

#### Scenario: Literal fixtures prove contract [r[provider-router-runtime-services.verification.literal-fixtures]]
- GIVEN model, stream, failure, usage, auth, and receipt fixtures are checked
- WHEN tests build expected data
- THEN expected provider-facing or neutral shapes MUST be explicit literals, not generated by calling the implementation under test

#### Scenario: Desktop parity remains intact [r[provider-router-runtime-services.verification.desktop-parity]]
- GIVEN desktop provider/router/auth adapters are used
- WHEN known-provider fail-closed, OpenAI Codex separation, credential-pool selection, and disabled embedded defaults are exercised
- THEN behavior MUST match existing provider/router contracts after projection through safe runtime DTOs

#### Scenario: Closeout validates provider services [r[provider-router-runtime-services.verification.closeout]]
- GIVEN implementation is complete
- WHEN focused validation runs
- THEN provider/runtime tests, embedded SDK checks if the SDK surface changed, Cairn validation/gates, and diff checks MUST pass
