# Design: Converge Provider Router Owners

## Concern ownership

Provider/router behavior should have one owner per concern:

| Concern | Owner rule |
|---|---|
| Provider-native body shaping | router backend or provider-native backend module |
| Routing, fallback, cooldown, and model selection | `clanker-router` |
| Auth refresh/probing and account discovery | provider-scoped router credential/auth modules |
| Clankers transcript/request compatibility | `clankers-provider` adapter bridge only |
| Cache-key projection from compatibility request | shared bridge helper with literal fixtures |
| Stream normalization | provider backend parser entrypoint with runtime seam tests |
| Compatibility error/event translation | `clankers-provider` adapter bridge |

Adapters may translate DTOs and errors, but they must not reconstruct provider-native JSON bodies, retry loops, entitlement probes, cooldown policy, or stream state machines.

## Migration approach

1. Generate an owner inventory from current constructor/helper sites.
2. Pick one duplicated concern with deterministic fixtures, such as cache-key request projection, provider request projection, or stream normalization.
3. Move policy to the declared owner or replace local logic with a call to the existing owner.
4. Keep compatibility fixtures literal: expected request/stream JSON must not be generated by the helper under test.

## Verification

Rails should fail closed when:

- a new `CompletionRequest` constructor misses a shared field;
- a compatibility adapter builds provider-native body JSON directly;
- local and routed adapters compute different cache/request projections;
- stream normalization is only covered by helper tests and lacks a parser-entrypoint seam test for touched provider streams.
