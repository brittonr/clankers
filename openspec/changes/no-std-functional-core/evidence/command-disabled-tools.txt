# Evidence: command disabled-tools parity

## Summary

- Cover accepted disabled-tool updates, filtered-tool rebuild application, stale-slot rejection, and feedback rejection through `crates/clankers-controller/src/command.rs`.
- Assert `CoreInput::SetDisabledTools` yields reducer output `ApplyToolFilter { effect_id, disabled_tools }`, the controller seam consumes that work through the rebuilder, and matching feedback clears the pending slot.
- Assert matching `ToolFilterApplied` feedback emits reducer-driven `DaemonEvent::DisabledToolsChanged`, and end-to-end command handling preserves `DaemonEvent::DisabledToolsChanged` before the success `DaemonEvent::SystemMessage` acknowledgement.
- Assert stale pending rebuild rejection preserves unchanged state, emits only error acknowledgement, and performs no rebuild shell work.
- Assert mismatched feedback yields typed `CoreError::ToolFilterMismatch`, out-of-order feedback yields typed `CoreError::OutOfOrderRuntimeResult`, both preserve unchanged state, emit only error acknowledgement, and emit no `DaemonEvent::DisabledToolsChanged`.
